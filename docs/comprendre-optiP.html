optiP — Sous le capot

Comprendre la technologie de l'app, sans être développeur

Ce guide explique en français simple ce qu'est optiP, comment ça marche, et qui sont les acteurs (Supabase, GitHub, Vercel). L'objectif : que tu puisses suivre une conversation technique et savoir ce que désigne chaque mot, sans pour autant savoir coder.

1 Qu'est-ce qu'optiP ?

optiP est une application web — comme un site internet, mais qui se comporte comme une vraie app sur ton iPhone ou ton iPad. Une fois installée sur l'écran d'accueil, elle ressemble à n'importe quelle app native, sauf qu'elle est servie par un serveur distant.

Son rôle : aider un pilote Air France à visualiser son planning, simuler sa paie, suivre son Article 81, comparer des scénarios de rotations, et garder tout ça accessible même sans réseau.

Analogie : optiP, c'est comme un tableur Excel très spécialisé pour pilotes, qui sait aller chercher tout seul tes rotations dans CrewBidd, qui se synchronise entre ton iPhone et ton iPad, et qui se met à jour automatiquement quand on corrige une formule.

2 Les trois mondes qui font tourner optiP

Toute app web moderne repose sur trois grandes briques. optiP suit ce schéma classique :

┌────────────────┐ ┌────────────────┐ ┌────────────────┐ │ GITHUB │ │ VERCEL │ │ SUPABASE │ │ (le code) │ → │ (l'hébergeur) │ ←→ │ (la data) │ │ │ │ │ │ │ │ Versionne │ │ Construit et │ │ Stocke les │ │ l'historique │ │ publie l'app │ │ données et │ │ du code │ │ sur internet │ │ authentifie │ └────────────────┘ └────────────────┘ └────────────────┘ ↑ │ ┌──────┴──────┐ │ TON iPAD │ │ (le client)│ └─────────────┘
  • GitHub garde une trace de chaque ligne de code écrite pour optiP, et de qui l'a modifiée et quand.
  • Vercel transforme ce code en site web accessible à l'adresse cm-offline.vercel.app.
  • Supabase joue à la fois le rôle de base de données (où sont stockées tes rotations, plannings, profils) et de serveur d'authentification (qui vérifie que c'est bien toi qui te connectes).
  • Ton iPad télécharge l'app depuis Vercel, garde une copie locale des données, et discute avec Supabase quand il a du réseau.
À retenir : ces trois briques sont des services payants à l'usage, hébergés ailleurs dans le monde. Tu ne possèdes pas de serveur — tu loues une part de très gros serveurs.

GitHub — La mémoire du code

Où le code vit, où l'historique se garde, où on collabore

1 C'est quoi GitHub ?

GitHub est un service en ligne (racheté par Microsoft en 2018) qui héberge le code source des projets logiciels. C'est l'équivalent d'un Google Drive, mais spécialisé pour les développeurs.

La différence : GitHub n'héberge pas que des fichiers, il héberge tout l'historique des modifications. Chaque caractère ajouté, supprimé ou déplacé est tracé, avec son auteur et sa date.

Analogie : imagine un grand cahier où chaque page contient le code de l'app à un instant précis. Chaque fois qu'on modifie une virgule, on ne gomme pas — on tourne la page et on recopie tout avec la modif. GitHub garde le cahier entier, indéfiniment.

2 Vocabulaire utile

MotSens
Repository (repo)Un projet. optiP a un repo qui contient tous ses fichiers.
CommitUne modification enregistrée. Chaque commit a un identifiant unique (6b2d9c7) et un message qui décrit ce qui a changé.
BranchUne ligne d'évolution du code. La branche principale s'appelle main.
PushEnvoyer des commits de ton ordinateur vers GitHub.
PullL'inverse — récupérer les commits depuis GitHub.
Pull Request (PR)Proposer de fusionner une branche dans une autre, avec discussion.

3 Comment c'est utilisé sur optiP

Quand on modifie une ligne de code dans optiP (ajout d'une fonctionnalité, correction d'un bug, refonte d'une page), voici le flux :

  1. La modif est faite localement, sur ton ordinateur.
  2. Un commit est créé avec un message du type fix gantt : bandeau MEP magenta opaque.
  3. Le commit est poussé vers GitHub avec la commande git push.
  4. GitHub conserve le commit dans l'historique de la branche main.
  5. Vercel (qu'on verra page suivante) est notifié automatiquement et déclenche un nouveau déploiement.

4 Pourquoi c'est précieux

  • Si une modif casse l'app, on peut revenir en arrière en un commit.
  • On voit qui a modifié quoi et pourquoi (le message du commit explique).
  • Le code est sauvegardé ailleurs que sur ton ordinateur : disque dur grillé, c'est pas grave.
  • Plusieurs personnes peuvent collaborer sans se marcher dessus.
Concrètement sur optiP : au démarrage de la conversation, on m'a indiqué que le dernier commit poussé sur GitHub est 6b2d9c7 revert swipe horizontal mois (Q1) — garde MEP opaque. Ça veut dire que la version en ligne de l'app reflète ce commit.

Vercel — L'hébergeur

Le service qui transforme ton code en site web accessible

1 C'est quoi Vercel ?

Vercel est une entreprise américaine fondée en 2015. Elle propose un service qui prend ton code, le compile (= prépare pour le navigateur), puis le publie sur internet à une adresse de ton choix. Vercel est aussi le créateur de Next.js, le framework qui structure optiP.

L'adresse publique d'optiP est https://cm-offline.vercel.app — c'est Vercel qui gère ce domaine.

Analogie : Vercel, c'est comme un imprimeur. Tu envoies un manuscrit (le code), il l'imprime joliment et il le met sur les présentoirs des kiosques du monde entier (les serveurs). Quand quelqu'un veut lire (= ouvrir l'URL), c'est lui qui lui tend le livre.

2 Le déploiement automatique

Vercel est connecté à GitHub. Dès qu'un commit est poussé sur la branche main, ça déclenche cette chaîne :

  1. Vercel détecte le push (notification GitHub).
  2. Vercel récupère le code à jour.
  3. Vercel installe les dépendances (les briques de code tierces, comme React).
  4. Vercel lance le build : next build — il transforme le code source en fichiers optimisés.
  5. Si tout réussit, la nouvelle version remplace l'ancienne en quelques secondes.
  6. Si le build échoue, l'ancienne version reste en ligne (rien ne casse).

3 Ce que Vercel fait en plus

  • CDN mondial : copies de ton site sur des serveurs partout dans le monde. Quand tu ouvres l'app depuis Paris, tu touches un serveur à Paris, pas un à San Francisco. Résultat : c'est rapide.
  • HTTPS automatique : le petit cadenas vert dans la barre d'adresse, c'est lui qui s'en occupe (gratuit, automatique).
  • Logs : si quelque chose plante côté serveur, Vercel garde des traces consultables.
  • Aperçus : pour chaque branche autre que main, Vercel peut publier une version d'essai temporaire.

4 Côté serveur vs côté client

Un point important pour comprendre la suite : il y a deux endroits où du code peut tourner.

CôtéOù ?Quoi ?
ServeurSur les machines de VercelLogique sensible, requêtes à Supabase avec clés secrètes, génération initiale des pages.
ClientDans ton navigateur SafariL'interface, les interactions, le rendu Gantt, le calcul de paie hors ligne.

Sur optiP, on a maximisé la part client pour pouvoir tout faire fonctionner hors ligne. Vercel sert surtout des fichiers statiques pré-construits et quelques fonctions serveur (les server actions).

Plan Vercel utilisé : le plan gratuit (Hobby) suffit pour optiP — c'est une app pour quelques utilisateurs, pas un site grand public.

Supabase — La base de données

Le coffre-fort des données et le gardien des comptes utilisateurs

1 C'est quoi Supabase ?

Supabase est un service en ligne qui propose tout ce dont une app a besoin côté serveur : base de données, authentification, stockage de fichiers, fonctions serveur. Il joue le rôle de "backend" pour optiP.

Au cœur de Supabase, il y a PostgreSQL, une base de données très réputée et open-source utilisée par énormément de gros services dans le monde.

Analogie : Supabase, c'est ta banque. Tes données (rotations, profils, A81) y sont rangées dans des coffres (= des tables). Pour y accéder, tu dois prouver ton identité (= login). Le banquier vérifie que tu n'as le droit de toucher qu'à tes propres coffres, pas à ceux du voisin.

2 Les tables d'optiP

Une table, dans une base de données, c'est l'équivalent d'un onglet Excel : des colonnes (les champs) et des lignes (les enregistrements). Voici les principales tables d'optiP :

TableCe qu'elle contient
user_profileTon profil paie : PVEI, KSP, base, fonctions, etc.
planning_itemChaque vol, congé, sim, repos planifié.
pairing_signatureLe détail d'une rotation type (legs, escales, durées).
pairing_instanceUne instance datée d'une rotation (départ le 15 janvier).
annexe_tableLes barèmes officiels : DDA, A81, IR/MF, etc.
user_notesTes notes libres sur des dates ou des scénarios.
auth_logHistorique des connexions (audit de sécurité).

3 Les migrations SQL

Quand on veut changer la structure d'une table (ajouter une colonne, par exemple), on n'édite pas la base directement. On écrit un fichier appelé migration SQL qui décrit le changement.

Dans optiP, ces fichiers sont dans le dossier supabase/migrations/. On en compte aujourd'hui plus de 40, numérotés 0001_init.sql jusqu'à 0040_auth_log_session_lost.sql. Chacun raconte une évolution de la base.

Pourquoi c'est utile : les migrations sont versionnées avec le code dans Git. Si on remet la base à zéro, on rejoue les migrations dans l'ordre et on retrouve exactement la même structure.

4 Row Level Security (RLS)

C'est la fonctionnalité qui protège tes données. Au lieu de faire confiance à l'app pour ne te montrer que tes lignes, on dit à PostgreSQL lui-même : « cet utilisateur n'a le droit de voir que les lignes dont la colonne user_id correspond au sien ».

Même si quelqu'un trouvait une faille dans l'app, il ne pourrait pas accéder aux données d'un autre utilisateur — la sécurité est au niveau de la base, pas seulement de l'app.

5 Authentification

Supabase gère aussi les comptes utilisateurs : email, mot de passe, sessions, tokens. Quand tu te logues, Supabase te délivre un jeton de session (un JWT) que ton iPad conserve. Ce jeton prouve à chaque requête que c'est bien toi.

La stack frontend

Next.js, React, TypeScript, Tailwind : les couches de l'app

Le "frontend" c'est tout ce que tu vois et avec quoi tu interagis dans l'app. Voici les outils qui le composent.

1 Next.js (le framework)

Next.js est un framework qui sert de squelette à l'app. Il gère le routage des pages (URL → page affichée), la performance, le rendu serveur, le pré-chargement.

optiP utilise la version 16 de Next.js (très récente, sortie fin 2025). C'est pour ça que le fichier AGENTS.md du projet précise : « This is NOT the Next.js you know » — les conventions ont évolué.

Analogie : Next.js, c'est comme la charpente d'une maison. Tu n'y touches pas tous les jours, mais elle décide où vont les murs, où sont les portes, et comment l'électricité circule.

2 React (la bibliothèque d'interface)

React est une bibliothèque créée par Facebook qui permet de construire l'interface utilisateur en assemblant des composants (boutons, cartes, listes, formulaires).

Chaque composant a un état (les données qu'il affiche) et se redessine automatiquement quand cet état change. Quand tu cliques sur "+ Nouveau profil", c'est un composant qui passe d'un état "fermé" à un état "ouvert".

3 TypeScript (le typage)

TypeScript est une version améliorée de JavaScript (le langage du web). Il ajoute des types : on déclare qu'une variable est un nombre, une chaîne de caractères, une date, etc. L'éditeur de code peut alors prévenir avant même de tester quand on essaie de faire une opération impossible.

Concrètement : ça évite des bugs du type « on a essayé de calculer une moyenne sur une date ». L'erreur est attrapée pendant l'écriture du code, pas dans l'app en production.

4 Tailwind CSS (le style)

Tailwind est un système de styles qui permet d'habiller l'app (couleurs, espacements, polices, ombres) en collant des étiquettes directement sur les éléments. Par exemple, className="bg-blue-500 text-white p-4 rounded-lg" rend un bouton bleu, texte blanc, padding moyen, coins arrondis.

L'offline — Le cœur d'optiP

PWA, Service Worker, IndexedDB : pourquoi ça marche sans réseau

La règle d'or d'optiP, c'est que tout doit fonctionner sans internet. Pas de WiFi en altitude, pas de 4G en escale lointaine : aucune importance. Voici comment c'est possible.

1 Une PWA, c'est quoi ?

PWA veut dire Progressive Web App. C'est un site web qui sait :

  • S'installer sur l'écran d'accueil de ton téléphone (comme une vraie app).
  • Fonctionner sans réseau (les données et le code sont en cache local).
  • Recevoir des notifications push (pas encore activé sur optiP, mais possible).

L'avantage : pas besoin de passer par l'App Store. La mise à jour se fait au prochain démarrage.

2 Le Service Worker

Le Service Worker est un mini-programme JavaScript qui tourne en arrière-plan dans ton navigateur, séparé de l'app. Il intercepte les requêtes réseau et décide :

  • « J'ai déjà la réponse en cache → je la renvoie sans demander à internet. »
  • « Pas en cache → je demande à internet ET je mets en cache. »
  • « Pas de réseau → je sers la version en cache. »

C'est lui qui permet à l'app de s'ouvrir instantanément même offline. Sur optiP, il est généré par Serwist.

Analogie : le Service Worker, c'est un majordome devant la porte de ton iPad. Quand l'app demande "donne-moi la page Gantt", il regarde d'abord dans son armoire (le cache). Si elle y est, il la sert tout de suite. Sinon, il va sonner chez le voisin (le serveur).

3 IndexedDB (la base locale)

Toutes les données dont l'app a besoin sont copiées dans une base de données dans ton iPad, appelée IndexedDB. Sur optiP, on l'utilise via une bibliothèque qui s'appelle Dexie (plus simple à manipuler).

Ce qui y est stocké :

  • Tes rotations (les vols catalogués pour les mois courants).
  • Tes plannings (scénarios A, B, C avec leurs vols, congés, sim).
  • Ton profil paie (toutes les versions).
  • Tes overrides A81 et tes notes.
  • Les barèmes (DDA, IR/MF, taux A81 par zone).

4 Le flux Push / Pull

Quand tu es offline et que tu modifies un vol, la modif part dans une file d'attente locale. Le badge orange sur le bouton Push compte les modifs en attente.

Quand tu retrouves le réseau :

  1. Push : tes modifs locales sont envoyées vers Supabase.
  2. Pull : optiP redemande à Supabase les rotations / plannings à jour pour les mois proches, et écrase la copie locale.

Le Pull est parallélisé (plusieurs mois en même temps) — gain de 27 secondes à environ 5 secondes pour 8 mois.

Risque connu : si tu modifies les mêmes données sur 2 appareils sans pousser entre les deux, il peut y avoir des conflits. Pour l'instant le dernier qui pousse gagne. Une vraie gestion de conflit est un chantier ouvert.

Le parcours d'une donnée

Du PDF EP4 jusqu'au calcul de paie affiché sur ton écran

Suivons concrètement le voyage d'un vol, depuis qu'il existe dans CrewBidd jusqu'à ce qu'il alimente ton estimation de paie. Cela permet de voir comment toutes les briques s'enchaînent.

1 Récupération depuis CrewBidd (le scraper)

CrewBidd est l'outil d'Air France qui publie les rotations disponibles chaque mois. optiP a un module appelé scraper qui se connecte à CrewBidd avec tes identifiants et télécharge les rotations dans la base Supabase.

Pour économiser les requêtes (règle stricte : environ 300 par mois maximum), le scraper ne télécharge que ce qui a changé depuis la dernière fois.

2 Stockage dans Supabase

Chaque rotation est éclatée en deux objets :

  • Une signature (pairing_signature) : le type de rotation, identique pour plusieurs occurrences. Ex. : un Paris-Tokyo avec ces escales, ces durées de vol, ces avions.
  • Des instances (pairing_instance) : les occurrences datées de cette signature. Ex. : Paris-Tokyo qui part le 5 janvier, le 12 janvier, etc.

3 Synchronisation vers ton iPad

Au prochain Pull (bouton ↻ dans la barre du haut), ces rotations sont copiées dans la base IndexedDB locale de ton iPad.

Tu peux maintenant les voir dans le Catalogue de l'app, même offline.

4 Placement dans ton planning

Tu choisis un vol dans le Catalogue (ou via la recherche sur une date) et tu le poses sur l'un des trois scénarios A / B / C. Cette opération crée un planning_item dans IndexedDB localement.

Au prochain Push, le planning_item est envoyé vers Supabase.

5 Calcul de paie en local

Quand tu regardes la colonne paie sur le Gantt, optiP fait tout le calcul dans ton iPad, sans aller au serveur :

  1. Il prend ton profil paie (PVEI, KSP, fonctions, etc.) pour le mois affiché.
  2. Il prend tous les planning_item du mois et leurs signatures associées.
  3. Il applique les formules : HCv, HCvr, PV, HS, MGA, BRUT, A81.
  4. Il croise avec les barèmes (taux A81 par zone, primes 1er mai, IR/MF, etc.).
  5. Il affiche le résultat.
Pourquoi tout en local : tu peux simuler un scénario complet à 35 000 pieds au-dessus de la Sibérie. Aucune requête réseau, et tes données ne quittent pas l'appareil tant que tu n'as pas Push.

6 EP4 : l'inverse

L'onglet EP4 compare ton planning estimé avec ton vrai bulletin EP4 (le PDF officiel). Tu importes le PDF, optiP le lit avec pdfjs-dist, en extrait les chiffres, et te montre les écarts ligne par ligne.

La vie d'une modif de code

Du clic dans la conversation à l'app à jour sur ton iPad

Tu me demandes "ajoute un bouton X" ou "corrige le calcul Y". Voici ce qui se passe ensuite, étape par étape.

┌──────────────────────────┐ │ 1. Tu me décris le besoin│ Conversation Claude └────────────┬─────────────┘ ▼ ┌──────────────────────────┐ │ 2. Je lis le code │ Tools Read, Grep, Glob │ concerné │ └────────────┬─────────────┘ ▼ ┌──────────────────────────┐ │ 3. J'écris la modif │ Tool Edit │ dans tes fichiers │ └────────────┬─────────────┘ ▼ ┌──────────────────────────┐ │ 4. On teste localement │ npm run dev │ si besoin │ └────────────┬─────────────┘ ▼ ┌──────────────────────────┐ │ 5. On commit + push │ git commit + git push └────────────┬─────────────┘ ▼ ┌──────────────────────────┐ │ 6. Vercel build + déploie│ ~1 à 2 min └────────────┬─────────────┘ ▼ ┌──────────────────────────┐ │ 7. Ton iPad voit la MAJ │ Point bleu sur la corbeille │ via Service Worker │ └────────────┬─────────────┘ ▼ ┌──────────────────────────┐ │ 8. Tu cliques corbeille │ Wipe cache + reload │ → nouvelle version │ └──────────────────────────┘

1 Le détail des étapes

  1. Lecture du code : j'explore les fichiers concernés pour comprendre l'existant avant de toucher quoi que ce soit.
  2. Modification : j'utilise des outils d'édition précis (remplacement de texte) pour ne pas casser ce qui marche.
  3. Vérification locale : on lance npm run dev pour ouvrir la version en cours de modif dans un navigateur. C'est l'équivalent d'un brouillon imprimé avant publication.
  4. Commit : on enregistre la modification avec un message clair. C'est une étape réversible.
  5. Push : on envoie le commit à GitHub.
  6. Vercel détecte le push, construit la nouvelle version et la met en ligne.
  7. Ton iPad, quand tu rouvres l'app, détecte qu'une nouvelle version existe (le point bleu apparaît sur l'icône corbeille de la barre du haut).
  8. Tu cliques sur la corbeille → l'ancien cache est vidé, l'app recharge la nouvelle version.
Ce qui rend ça fluide : aucune étape ne demande d'aller sur le serveur, d'installer quoi que ce soit en SSH, ou de manipuler une base de données à la main. Tout est automatisé par les outils.

2 Quand la base doit changer

Si la modif demande aussi une évolution de la base Supabase (ex. ajouter une colonne à user_profile), il faut en plus écrire une migration SQL et l'appliquer manuellement via le tableau de bord Supabase (ou via la CLI Supabase).

C'est pour ça que tu vois 40+ fichiers numérotés dans supabase/migrations/.

Sécurité et données privées

Qui voit quoi, où sont stockées tes données, comment elles sont protégées

1 Où sont tes données ?

EndroitQuoiSécurité
Sur SupabaseToutes tes données serveur (rotations, planning, profil, A81)HTTPS + RLS Postgres
Sur ton iPadCopie locale dans IndexedDBIsolée à ton compte iOS
Sur GitHubLe code, jamais tes donnéesRepo privé
Sur VercelLe code construit, des logs serveurHTTPS
Important : tes données métier (planning, paie) ne sont jamais dans le code source. Le code dit "comment afficher" et "comment calculer", mais pas "voici les rotations de Mickaël".

2 Comment ton identité est vérifiée

  1. Tu saisis email + mot de passe.
  2. Supabase vérifie que c'est correct et te renvoie un token JWT (un long code chiffré).
  3. Ce token est stocké dans Safari (cookie + localStorage).
  4. À chaque requête vers Supabase, le token est envoyé. Supabase l'inspecte et sait que c'est toi.
  5. Si le token expire, ton iPad demande automatiquement un nouveau (refresh token), tant que la session est valide.

3 La whitelist

optiP n'est pas ouvert à n'importe qui. Une table whitelist contient les emails autorisés. Un inconnu qui tente de créer un compte avec un email non listé est refusé.

Tu es administrateur (is_admin = true) : tu peux gérer cette whitelist, déclencher des actions sensibles (release, snapshots, scrape), et accéder à des pages d'admin que les autres ne voient pas.

4 Sauvegarde et restauration

  • Supabase fait des backups automatiques de la base (selon le plan).
  • Il existe une fonction d'export (snapshot JSON) qu'on peut télécharger en admin.
  • Il existe une fonction d'import CSV pour recharger des mois entiers depuis les extracts d'origine.

5 En cas de perte d'iPad

  1. Tu te logues sur un nouvel appareil avec ton email + mot de passe.
  2. Tu cliques sur le nuage circulaire (Pull).
  3. Tout ton historique est récupéré depuis Supabase.

La seule chose qui pourrait être perdue : les modifs faites offline avant la perte de l'appareil, si tu n'avais pas pu Push.

Bonne pratique : avant un long trip offline, fais un Push si tu as des modifs en attente. Au retour, fais un Pull pour redescendre les MAJ serveur (collègue qui a complété un mois, scraper qui a tourné).

Migration SQLFichier qui décrit un changement de structure de la base.
RLSRow Level Security. Règle qui filtre les lignes selon l'utilisateur connecté.
RPCRemote Procedure Call. Une fonction qu'on appelle côté serveur depuis le client. (Dans optiP, RPC désigne aussi la Période de Repos Compensateur — c'est un sigle métier homonyme.)

4 Client et offline

Service WorkerMini-programme qui intercepte les requêtes réseau et sert le cache local.
IndexedDBBase de données embarquée dans le navigateur.
DexieBibliothèque qui simplifie l'usage d'IndexedDB.
CacheCopie locale rapide d'éléments téléchargés depuis le serveur.
JWTJSON Web Token. Le "passeport" chiffré qui prouve qui tu es.

Le mot de la fin : tu n'as pas besoin de tout savoir coder pour piloter intelligemment le développement d'optiP. Comprendre quelles briques font quoi suffit à formuler des demandes précises, à anticiper les coûts, et à juger ce qui est faisable ou coûteux à changer.
